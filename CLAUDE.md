# TREETEX

## О проекте

TREETEX — коммерческая платформа нового поколения, сочетающая возможности TikTok + интернет-магазина + маркетплейса + live shopping. Пользовательские роли: покупатель, продавец, блогер, компания, администратор. Полное техническое задание — в [PLAN.md](PLAN.md).

## Текущий статус

Проект в разработке. Изначально был проработан как набор интерактивных дизайн-прототипов (`.dc.html` + `support.js`) в проприетарном формате дизайн-инструмента. Сейчас ведётся переход к реальной технической реализации.

## Стек

- **Frontend**: Next.js (App Router) + React 18 + Tailwind CSS — `apps/web`
- **Backend**: NestJS + TypeORM — `apps/api`
- **База данных**: PostgreSQL 16 — сервис `db`
- **Общие типы**: TypeScript-интерфейсы (Product, User, Order, Stream) — `packages/shared`
- **Инфраструктура**: Docker Compose (`docker-compose.yml`), три сервиса: `web` (порт 3000), `api` (порт 4000), `db` (порт 5432), dev-режим с hot reload через volume-mount

## Структура репозитория

```
/treetex
  docker-compose.yml       — оркестрация всех сервисов
  PLAN.md                  — техническое задание (полное описание продукта)
  CLAUDE.md                — этот файл
  /history                 — история значимых решений и изменений (см. ниже)
  /apps
    /web                   — Next.js frontend
    /api                   — NestJS backend
  /packages
    /shared                — общие TypeScript-типы для web и api
  *.dc.html, support.js    — исходные дизайн-прототипы (референс для переноса вёрстки/контента, не деплоятся)
  /uploads                 — референсные материалы (лого и т.п.)
```

## Правило: история значимых решений

**После каждого plan mode, завершённого через `ExitPlanMode` (то есть план был одобрен пользователем), в папку `/history` добавляется отдельный новый файл**, фиксирующий, что и почему было решено/сделано.

- Формат имени: `/history/NNN-краткое-описание.md`, где `NNN` — порядковый номер с ведущими нулями (001, 002, 003…), а `краткое-описание` — 2–5 слов через дефис на русском или английском, отражающих суть изменения.
- Номер берётся как `(номер последнего существующего файла в /history) + 1`. Никогда не переиспользовать и не менять задним числом номера уже существующих файлов.
- Каждый файл — самостоятельная запись, не дополнение к предыдущей.
- Не удалять и не переписывать прошлые записи истории — это журнал, а не живой документ.

### Структура записи в истории

```markdown
# NNN — Краткое название

Дата: YYYY-MM-DD

## Контекст
Что было до этого решения, какая задача/проблема стояла.

## Решение
Что именно решили и почему (в том числе отвергнутые альтернативы, если обсуждались).

## Изменения
Конкретные файлы/сервисы/зависимости, которые были созданы/изменены.

## Результат проверки
Как было подтверждено, что изменение работает (если применимо).
```

Триггер — именно факт прохождения через `ExitPlanMode` (одобренный план), а не любое мелкое редактирование файла вне режима плана.
