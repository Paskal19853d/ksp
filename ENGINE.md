# TREETEX ENGINE — архитектура backend-движка

> Этот файл описывает устройство и правила построения backend-"движка" TREETEX при целевом масштабе **сотни тысяч – миллионы пользователей** в первый-второй год. Он не дублирует функциональные требования ([PLAN.md](PLAN.md)) и чек-листы безопасности/структуры ([SKILL.md](SKILL.md)) — ссылается на них. Читается наравне с CLAUDE.md/PLAN.md/SKILL.md в начале каждой сессии.

## Текущее состояние (на момент написания)

Backend реализован **только для авторизации**: `apps/api/src/auth/` + `apps/api/src/users/` (JWT в httpOnly cookie, bcrypt, rate limiting через `@nestjs/throttler`, глобальный `ValidationPipe`). База — Postgres 16 с единственной таблицей `users`, `synchronize: true` (dev-режим). Все остальные сущности (товары, заказы, магазины, эфиры, чат, модерация, CMS) — mock-данные в клиентском React state, backend их не обслуживает.

Разделы ниже описывают целевую архитектуру и порядок движения к ней — не то, что уже построено.

---

## 1. Общая форма архитектуры

**Модульный монолит на NestJS. Не микросервисы.**

Один деплоящийся NestJS-процесс с чётко разделёнными внутренними модулями (см. §8), горизонтально масштабируемый через несколько stateless-реплик за балансировщиком. Один кластер PostgreSQL.

**Почему не микросервисы:** микросервисы решают организационную проблему (независимые деплои для независимых команд) ценой распределённых транзакций, сетевых вызовов вместо вызовов функций, N пайплайнов вместо одного. У соло-разработчика нет второй команды, от которой нужно отделяться — платить за это нечем компенсировать. Модульный монолит с грамотным кэшированием и пулингом БД масштабируется горизонтально до миллионов пользователей без этой цены.

**Почему не бесструктурный монолит:** товары/заказы/эфиры/чат/модерация/финансы/CMS имеют разный профиль нагрузки (real-time и эфемерное — чат, счётчики зрителей; транзакционное и durable — заказы, финансы; read-heavy и кэшируемое — CMS, каталог). Чёткие модульные границы с самого начала (см. §8) делают возможным будущее выделение сервиса (если реально понадобится) без переписывания.

## 2. Слой данных (PostgreSQL)

**Сделать немедленно, до Products-модуля:**
- Выключить `synchronize: true`. Ввести миграции через встроенный TypeORM tooling (`typeorm migration:generate`/`migration:run`) — уже в зависимостях, читает entity-метаданные. Хранить в `apps/api/src/migrations/` (папка уже предусмотрена в SKILL.md).
- Каждая новая entity — с `@Index()` на реальных паттернах запросов (foreign keys, `slug`, `status`, `createdAt` для ленты) сразу при создании — дёшево сейчас, дорого на заполненной таблице под нагрузкой.
- **PgBouncer** (transaction-pooling) перед Postgres — как только появится больше одного инстанса api. Если используется managed Postgres с built-in пулингом (Neon, Supabase, RDS Proxy) — использовать его, не поднимать свой.

**Отложено — не строить заранее:**
| Решение | Триггер пересмотра |
|---|---|
| Read-реплики | Устойчивая нагрузка на CPU/read-IOPS primary выше ~60-70% на пике, или деградация p95 read-latency именно в read-heavy периоды — после того, как кэширование (§3) уже внедрено и запросы проиндексированы. Первая реплика — под аналитику/admin-дашборды (наименее latency-чувствительный трафик), не под user-facing чтение. |
| Шардинг | Write-throughput одной таблицы упирается в потолок полностью оттюненного primary с offload на реплики — реалистично только для write-heavy таблиц вроде `stream_chat_messages`. И даже тогда сначала проверить, что этим данным вообще место в Postgres, а не в Redis/стрим-логе (см. §3) — это чаще снимает нужду в шардинге, чем создаёт. |
| Отдельный OLAP/warehouse | Admin/seller-дашборд запросы измеримо замедляют checkout-путь — тогда решение: materialized view с ночным рефрешем или отдельная read-реплика под аналитику, не новый продукт для данных. |

Никакой мультирегиональной БД, CQRS/event-sourcing фреймворка сейчас не нужно.

## 3. Кэширование и real-time (Redis)

**Redis — единая точка** для: кэша (лента, каталог, CMS-страницы), rate-limiting между инстансами (in-memory throttler ломается при горизонтальном масштабировании — каждый инстанс считает лимиты отдельно), pub/sub для live-чата и счётчиков зрителей, очередей (BullMQ). Managed Redis (Upstash / ElastiCache / Memorystore) — не самохостинг стейтфулного сервиса, от которого зависит всё остальное.

### Live-shopping real-time (PLAN.md называет это "самой важной частью проекта")

- **Транспорт — WebSocket, не SSE, не polling.** Нужна двунаправленность (чат туда-обратно, "купить не покидая эфир" с быстрым подтверждением). `@nestjs/websockets` + `socket.io`.
- **Горизонтальное масштабирование сокет-слоя — `socket.io-redis-adapter`.** Без него клиент на инстансе A не получит сообщение от клиента на инстансе B — один эфир окажется привязан к одному инстансу. Redis-адаптер публикует события через Redis, любой инстанс может доставить любому подключённому клиенту.
- **Счётчик зрителей** — Redis-счётчик/set (`INCR`/`DECR` при connect/disconnect), не запрос к Postgres на каждый рендер. Периодический снапшот (каждые 30-60 сек) в Postgres для пост-эфирной аналитики (уже есть mock-форма `liveHistory`/`liveKpis` в `seller.ts`).
- **Чат** — публикация в Redis pub/sub канал (или Redis Streams, если нужен replay при подключении) по ID эфира. Персист в Postgres — асинхронно/батчами, не блокируя fan-out на каждое сообщение.
- **⚠️ Инвариант, который нельзя нарушать: списание остатков товара во время эфира — ТОЛЬКО через ACID-транзакцию в Postgres** (`SELECT ... FOR UPDATE` или атомарный `UPDATE products SET stock = stock - 1 WHERE id = $1 AND stock > 0`, проверка affected rows). Redis может кэшировать отображаемое значение для быстрого UI-обновления, но никогда не является источником истины для самого списания — это ровно та же гарантия, что и в обычном checkout, здесь нет отдельной модели консистентности. После успешной покупки — broadcast обновлённого остатка всем зрителям эфира через тот же WebSocket-канал.
- **Фоновые задачи — BullMQ** (на том же Redis) для payout-батчей, email-уведомлений, вебхуков транскодирования от видео-провайдера, батчинга персиста чата.

**Почему не Kafka/RabbitMQ сейчас:** Redis pub/sub + Socket.IO adapter достаточно для fan-out чата/присутствия на этом масштабе и на порядок проще в эксплуатации, чем Kafka-кластер. Kafka оправдана при потребности в durable, replayable, multi-consumer event-логах с строгим порядком — этой потребности пока нет.

## 4. Видео / Live-стриминг

**Рекомендация: управляемый сервис, не самостоятельный RTMP/HLS.**

Основной вариант — **Cloudflare Stream** (ingest + adaptive HLS transcoding + встроенный CDN + запись, usage-based биллинг; консолидируется с CDN для медиа из §6 — один провайдер). Запасной — **Mux** (более зрелый live-продукт, отличные вебхуки, более подробная аналитика зрителей "из коробки" — полезно для live-статистики).

Рассмотрены и отклонены как не первый выбор: **AWS IVS** (стоит sanity-check в bake-off из-за built-in low-latency mode и IVS Chat, если инфраструктура окажется AWS-центричной), **Agora/LiveKit** (заточены под many-to-many видеозвонки, здесь один-ко-многим broadcast — избыточная сложность).

**Почему не самостоятельная инфраструктура (nginx-rtmp, Ant Media, Owncast):** это отдельная инженерная специализация — geo-распределённый ingest, транскодирование в несколько битрейтов, HLS-дистрибуция (= свой CDN), запись VOD. Для соло-разработчика, чья реальная дифференциация — commerce/chat-слой поверх видео, а не сама видео-труба — это самое рискованное место потратить время на этом плане.

**Как встраивается:** NestJS-модуль `streams` владеет только метаданными (кто в эфире, какие товары закреплены, время начала, снапшоты зрителей) и вызывает API провайдера для выдачи ingest-креденшлов + принимает вебхуки о жизненном цикле эфира. Видео-байты никогда не проходят через свои серверы — это провайдер-агностичный дизайн, upgrade/смена провайдера позже — контролируемая миграция (замена ingest URL и плеера), не переписывание.

## 5. Поиск

**Сейчас: Postgres full-text search (tsvector/tsquery + GIN-индекс).** Покрывает поиск по товарам/продавцам/категориям/пользователям при правильном индексировании до десятков миллионов строк с суб-100мс латентностью — без новой инфраструктуры, без отдельного индекса, который нужно держать в синхроне.

**Отложено:** Elasticsearch/Meilisearch/Typesense. Триггер — размер каталога, при котором даже оттюненный Postgres FTS деградирует, ИЛИ потребность в faceted-поиске (цена+категория+рейтинг+наличие одновременно, быстро, при большом каталоге). Когда триггер сработает — **Typesense или Meilisearch раньше Elasticsearch**: оба проще в эксплуатации и заточены именно под "мгновенный поиск по каталогу", а не под общецелевой log/analytics поиск, как Elasticsearch.

## 6. Хранилище медиа (фото товаров, аватары, VOD)

**Объектное хранилище + CDN, S3-совместимый API.** Рекомендация — **Cloudflare R2** (нулевая плата за egress — критично для медиатяжёлого приложения, где каждый скролл ленты повторно тянет изображения; консолидация с Cloudflare Stream из §4). Альтернатива при AWS-центричном стеке — **S3 + CloudFront**. **MinIO самохостинг — не рекомендуется**: возвращает ровно ту операционную нагрузку (бэкапы, доступность стейтфулного сервиса), ради устранения которой managed-хранилище и существует.

**Паттерн загрузки (реализует требование SKILL.md — "MIME+размер валидированы, отдаётся не из репозитория/локального диска"):**
1. Клиент запрашивает pre-signed upload URL у модуля `media` — эндпоинт валидирует заявленный MIME-тип и размер против allowlist **до** выдачи подписанного URL.
2. Клиент грузит файл напрямую в бакет по этому URL — байты не проходят через NestJS-сервер (важно: держит API stateless и не создаёт bandwidth-бутылочное горлышко для видео).
3. По завершении — callback/webhook записывает итоговый URL за сущностью (`Product.imageUrl`, `User.avatarUrl`). Для файлов выше базового порога — повторная проверка реального содержимого (magic bytes, не только заявленный MIME) фоновой BullMQ-задачей.
4. Все отдаваемые URL — CDN-адреса объектного хранилища, никогда не `apps/web/public` и не локальный диск API-контейнера (тем более при горизонтально масштабированных эфемерных контейнерах — локальный диск даже не был бы общим между репликами).

## 7. CMS

**Рекомендация: собственный небольшой модуль `cms` в NestJS. Не Strapi/Payload/Directus.**

Реальный охват CMS в TREETEX (по текущему mock `CmsPage{id,title,slug,updated,published}` и PLAN.md) — несколько статичных страниц (`/about`, `/terms`, `/privacy`, `/seller-rules`, промо-страницы) плюс баннеры/реклама. Поднимать headless CMS (отдельное приложение со своей БД, admin UI, циклом обновлений безопасности) ради управления пятью-дюжиной простых страниц — несоразмерная операционная нагрузка для соло-разработчика: пришлось бы либо дублировать управление пользователями/ролями в неё, либо строить интеграционный слой — реальная цена ради маргинальной пользы здесь.

**Конкретный дизайн:**
- `apps/api/src/cms/` по шаблону auth/users: `PageEntity` (`id`, `title`, `slug` unique+indexed, `status: 'draft'|'published'`, `updatedAt`, `authorId`) + колонка `content: jsonb` — массив типизированных блоков (`{type:'heading',text}`, `{type:'richtext',html}`, `{type:'image',url,alt}`) вместо цельного HTML-блоба. Это та же идея, что "content modeling" в headless CMS, но заточенная под реальные нужды TREETEX и в той же базе (не нужно отдельно бэкапить/синхронизировать).
- Admin-редактор — форма со списком блоков (добавить/переставить/редактировать/удалить) в существующей `apps/web/app/(admin)/cms/` — не drag-and-drop visual builder, это соответствует реальной задаче (страницы условий, не self-serve визуальный конструктор для маркетинг-команды).
- Публичный рендеринг — Next.js динамический роут по `slug`, серверный fetch опубликованной `PageEntity`, рендер блоков в HTML. Хороший кандидат на агрессивное кэширование (CDN/ISR — контент меняется редко относительно трафика).
- Rich-text блоки — через ту же XSS-санитизацию, что SKILL.md уже требует для пользовательского контента (чат, отзывы), даже несмотря на то, что авторы CMS — админы (defense in depth).
- Баннеры/реклама (`AdminBanner`, `AdminAd`) — отдельные entities в этом же модуле или соседнем `marketing` (свои поля: клик-трекинг, бюджет/расход, ссылки) — не смешивать со "страницами".

**Когда пересмотреть:** реальная потребность в сложных редакторских workflow (многошаговое утверждение, множество типов контента, self-serve для нетехнической команды) — не подтверждено ничем в PLAN.md сегодня.

## 7.1. Правило: доступ к данным на фронтенде — только через обёртки, никогда напрямую полем объекта

**Компонентам (`.tsx`) запрещено читать бизнес-поля сущностей напрямую** (`product.price`, `order.sum`, `product.rating`) и запрещено обращаться к `fetch`/`api.get(...)` напрямую внутри компонента. Вместо этого — два слоя обёрток в `apps/web/lib/data/`:

1. **Data-хуки** (`useProducts`, `useProduct`, `useOrders`/`useMyOrders`/`useSellerOrders`, `useReviews` и т.д.) — единственное место, которое обращается к `api.get/post/patch`. Компонент получает уже загруженные данные + `loading`/`error`, никогда сам не решает, откуда они берутся.
2. **Функции-форматтеры/маппер** (`toCardData`, `productImgUrl`, `productBadge`, `orderStatusLabels`) — единственное место, которое знает **как** сырое поле сущности превращается в отображаемое значение (форматирование цены, URL картинки, перевод статуса на человеческий язык, вычисление badge/скидки). Компонент вызывает `toCardData(product)` и рендерит результат — не пересчитывает `Math.round((1 - price/compareAtPrice) * 100)` инлайн в JSX.

**Почему:** это тот самый принцип, который в дизайн-прототипах (`.dc.html` + `support.js`) обеспечивал `{{price}}`-подстановки без завязки разметки на конкретную структуру данных. В реальном коде роль `{{ }}`-шаблонизатора играют TypeScript-функции и хуки — если цена, статус заказа или URL картинки задаются/форматируются в двух местах по-разному, они рано или поздно разойдутся (уже случалось: `imgUrl(seed)` для мока vs `productImgUrl(product)` для реальных данных — страницы, забытые при переходе на API, продолжали дергать старую функцию с `product.seed`, которого больше нет).

**Как применять:**
- Новая сущность с backend → сразу заводить `use<Entity>`/`use<Entity>s` хук в `apps/web/lib/data/use<Entity>.ts`, даже если пока только один компонент его использует.
- Любое вычисляемое/форматированное представление поля (цена в валюте, дата в локали, процент скидки, статус на человеческом языке, URL картинки) → отдельная экспортируемая функция рядом с хуком, не инлайн-выражение в JSX.
- Если то же вычисление нужно в двух компонентах — это сигнал, что оно уже должно было быть обёрткой, а не повод скопировать выражение второй раз.
- Ревью нового компонента: если в JSX встречается `.` сразу после названия сущности с бизнес-полем (`product.price`, `order.status`) вне вызова обёртки — это нарушение правила.

## 8. Границы модулей NestJS

Шаблон для каждого модуля ниже — **точно как в `apps/api/src/auth/` и `apps/api/src/users/`**: `<feature>.module.ts` (imports/providers/controllers/exports), `<feature>.controller.ts` (или `.gateway.ts` для WebSocket-модулей), `<feature>.service.ts`, `entities/`, `dto/`, `guards/`/`strategies/` только там, где модуль сам вводит новую auth-логику.

| Модуль | Владеет | Примечание |
|---|---|---|
| `users` *(есть)* | Аккаунты, роли | |
| `auth` *(есть)* | Login/register/JWT/guards | |
| `products` | Каталог, категории, вариации/размеры/цвета, остатки | Ядровая сущность — остальные модули ссылаются по ID, не лезут во внутренности |
| `sellers` | Профили магазинов, команда (`TeamRole`: Owner/Manager/Operator/Analyst — history/004), подписчики | "Компания" = продавец + команда, не отдельная роль |
| `orders` | Корзина, checkout, жизненный цикл заказа, возвраты | Транзакционное ядро; списание остатков — здесь (или вызывается из `streams`) |
| `reviews` | Отзывы/рейтинги товаров и продавцов | |
| `streams` | Метаданные live-эфиров, расписание, снапшоты зрителей, привязка товаров | Gateway-модуль, интеграция с вебхуками видео-провайдера (§4) |
| `chat` | Live-чат, комментарии к видео | Gateway-модуль, на Redis pub/sub (§3) |
| `notifications` | Лента уведомлений, (позже) push/email | Слушает доменные события других модулей, а не импортируется напрямую всеми |

| `moderation` | Очереди модерации видео/товаров/комментариев, жалобы | Ссылается на цели в `products`/`chat`/`streams`, владеет статусами решений |
| `cms` | Статичные страницы (блочный контент) | §7 |
| `marketing` | Баннеры, реклама | Соседний с `cms` |
| `media` | Выдача pre-signed upload, валидация | §6 |
| `search` | Postgres FTS запросы | Тонкий модуль сейчас — точка будущей замены на Typesense/Meilisearch (§5) |
| `admin` | Кросс-модульная агрегация для админки (дашборд, настройки платформы) | Компонует чтения из других модулей |

**Почему это позволяет будущее выделение сервиса без переписывания:** каждый модуль обращается к другим только через их публичные service-методы (никогда напрямую к чужому репозиторию/entity). Если `streams`/`chat` когда-нибудь реально понадобится масштабировать отдельно (самый вероятный кандидат — резко иной профиль нагрузки: bursty, connection-heavy, real-time) — извлечение в отдельный деплой станет механической заменой in-process вызовов на HTTP/RPC, а не распутыванием.

## 9. Дорожная карта реализации

Порядок — по разблокируемым фичам из PLAN.md на единицу усилия + порядку зависимостей:

1. **Миграции** (предпосылка, не фича) — выключить `synchronize`, завести `apps/api/src/migrations/`, закоммитить baseline-миграцию для `users`.
2. **Products** — разблокирует ленту/каталог/поиск/категории покупателя, карточку товара, CRUD товара продавца (создание/редактирование/фото/остатки/вариации). Самая большая единая разблокировка из PLAN.md.
3. **Media** — вместе с Products (загрузка фото товара — часть флоу создания товара, нет mock-заменителя для upload-пайплайна).
4. **Orders/Cart** — корзина, checkout, история заказов, управление заказами продавца, возвраты.
5. **Reviews** — небольшой, зависит от Products+Orders (verified purchase).
6. **Seller dashboard реальный API** — статистика/доход/продажи вместо mock `dashboardByPeriod`/`topSellerProducts`, теперь на реальных Orders/Products.
7. **Real-time слой: Redis + `streams` + `chat`** — самый крупный инфраструктурный рывок (Redis, WebSocket gateway, интеграция видео-провайдера). После коммерческого ядра, так как live-checkout зависит от реальных Orders/Products. Разблокирует явно обозначенную "самую важную часть проекта".
8. **Search** — Postgres FTS по реальным данным Products/Sellers/Streams.
9. **CMS + Marketing** — ниже приоритет коммерции/live, но прямолинейно и низкий риск после того, как паттерн модулей устоялся.
10. **Moderation** — очереди модерации, зависит от существования Products/Streams/Chat как целей модерации.
11. **Finance** — комиссии/выплаты, зависит от Orders (GMV) и Sellers.
12. **Blogger** — партнёрские ссылки/выплаты, можно относительно независимо/позже.
13. **Notifications** — можно начать как простая in-app лента, питаемая доменными событиями Orders/Streams/Reviews; позже — push/email через BullMQ.

## 10. Что не строить сейчас

| Не строить | Триггер пересмотра |
|---|---|
| Kubernetes | Потребность в сложной multi-service оркестрации/сетевых требованиях, которые managed PaaS (Render/Railway/Fly.io/ECS Fargate) не может выразить — маловероятно на этом масштабе, бутылочное горлышко раньше будет в data-слое (§2), не в оркестрации compute |
| Микросервисы | Организационный триггер (несколько независимых команд), не пользовательский масштаб |
| Шардинг Postgres | Конкретный измеренный потолок write-throughput на оттюненном primary — не "у нас много юзеров" |
| Read-реплики заранее | Измеренная нагрузка/латентность primary после того, как кэш уже внедрён — не превентивный прогрев |
| Elasticsearch | Размер каталога/faceted-поиск, которые Postgres FTS больше не тянет — и даже тогда сначала Typesense/Meilisearch |
| Своя видео-инфраструктура | Практически никогда для этой команды — см. §4 |
| Headless CMS | Реальная потребность в сложных мультиредакторских workflow — не "нужно несколько статичных страниц" |
| MinIO / самохостинг object storage | Managed-хранилище снимает целый класс операционной нагрузки за небольшую цену на этом масштабе |
| Kafka / тяжёлый брокер | Redis pub/sub + BullMQ покрывает fan-out и очереди; Kafka — для durable multi-consumer event-логов, которых пока нет |
| Мультирегиональный деплой | Демонстрированная потребность обслуживать географически далёких пользователей с низкой латентностью — и даже тогда CDN (уже в плане для видео/медиа) решает read-половину проблемы без мультирегиональности самого origin |
