## SKILL 1 — БЕЗОПАСНОСТЬ: проверка от взлома и утечки данных

### Применять к каждому новому endpoint, форме, компоненту с пользовательским вводом

#### Backend (NestJS, `apps/api`)

| Проверка | Что делать |
|-----------|-----------|
| SQL Injection | Только параметризованные запросы через TypeORM (`QueryBuilder`/repository methods с параметрами). Никогда не собирать SQL конкатенацией строк с пользовательским вводом |
| Валидация входных данных | Каждый DTO — через `class-validator`/`class-transformer`, `ValidationPipe` включён глобально |
| XSS | Все пользовательские текстовые поля (отзывы, чат, названия магазинов) — санитизировать перед сохранением/выводом, не доверять клиенту |
| SSRF | Любые URL от пользователя (аватар, обложка магазина, внешние ссылки) — валидировать, блокировать приватные диапазоны (`127.x`, `10.x`, `172.16-31.x`, `192.168.x`, `0.0.0.0`) |
| Auth на защищённых endpoint'ах | Каждый `/api/admin/*` и `/api/seller/*` — через Guard (`@UseGuards(AuthGuard, RolesGuard)`), проверка роли пользователя |
| Rate limiting | Публичные endpoint'ы — через `@nestjs/throttler`, базовый лимит на IP/пользователя |
| Brute-force | Auth endpoints (`/api/auth/login`): лимит попыток + временная блокировка по IP+email |
| Секреты в коде | Только `.env` (не коммитить). Никогда не хардкодить пароли, ключи, JWT-секреты, DB credentials в исходниках |
| Загрузка файлов (фото/видео товаров) | Проверять MIME type и размер, хранить вне `apps/web/public` без санитизации имени, отдавать через CDN/object storage, не из репозитория |
| Пароли | Только хешированные (bcrypt/argon2), никогда в открытом виде ни в БД, ни в логах |

#### Frontend (Next.js, `apps/web`)

| Проверка | Что делать |
|-----------|-----------|
| Токены авторизации | Не хранить JWT/сессионные токены в `localStorage`/`sessionStorage` — использовать httpOnly cookie или защищённое хранилище |
| XSS через `dangerouslySetInnerHTML` | Не вставлять пользовательский контент напрямую; если необходимо — только через санитайзер (например, `dompurify`) |
| CSRF | Мутирующие запросы (POST/PUT/DELETE) к API — с учётом cookie-based auth и CORS-политики на стороне NestJS |
| Open redirect | Проверять, что redirect-параметры (после логина, после checkout) ведут только на относительный путь или собственный домен |
| Console.log секретов | Не логировать токены, пароли, персональные данные покупателей в консоль браузера |
| Переменные окружения | `NEXT_PUBLIC_*` — только для несекретных значений (например, `NEXT_PUBLIC_API_URL`); секреты не должны попадать в клиентский бандл |

#### Чек-лист перед каждым новым модулем/endpoint'ом
```
[ ] Все inputs — валидированы через DTO/class-validator
[ ] SQL/TypeORM запросы — параметризованы, без конкатенации
[ ] Admin/seller endpoint — защищён Guard'ом с проверкой роли
[ ] Rate limit — настроен для публичных endpoint'ов
[ ] Токены — не в localStorage, а в httpOnly cookie
[ ] Секреты — в .env, не в коде и не в git
[ ] Файлы (фото/видео товаров) — проверены MIME + размер
[ ] Redirect — только относительный или собственный домен
```

## SKILL 2 — АДАПТИВНЫЙ ДИЗАЙН: от телефона до десктопа

### Применять к каждой новой UI-секции, компоненту, странице в `apps/web`

Согласно ТЗ (см. [PLAN.md](PLAN.md)), обязательны отдельные, полностью адаптирующиеся версии интерфейса:

#### Брейкпоинты проекта (Tailwind)

```
Mobile   : ≤ 640px   (sm)
Tablet   : 641–1024px (md/lg)
Laptop   : 1025–1440px (lg/xl) — целевая ширина 1440
Desktop  : 1441px+     (xl/2xl) — целевая ширина 1920+
```

Использовать стандартные Tailwind-брейкпоинты (`sm:`, `md:`, `lg:`, `xl:`, `2xl:`) как основной механизм адаптива — это уже включено в `apps/web/tailwind.config.ts`. Не изобретать собственные `@media` без необходимости.

#### Обязательные правила

**1. Mobile-first.** Базовые классы — под мобильный экран, модификаторы (`md:`, `lg:`) — для расширения на больших экранах.

**2. Тексты — предпочтительно относительные/clamp-размеры для заголовков:**
```css
font-size: clamp(20px, 3vw, 36px); /* заголовки */
```
Обычный текст — фиксированные Tailwind-классы (`text-base`, `text-sm`) допустимы, если проходят проверку на мобильном.

**3. Контейнеры — max-width + отступы:**
```html
<div class="w-full max-w-[1400px] mx-auto px-4 md:px-8 lg:px-12">
```

**4. Изображения товаров/видео — всегда адаптивные:**
```html
<img class="max-w-full h-auto" ... />
<!-- Фото товаров: object-fit: contain, не cover — не обрезать значимые детали -->
```

**5. Grid/Flex — авто-адаптация карточек товаров/видео:**
```css
grid-template-columns: repeat(auto-fill, minmax(240px, 1fr));
```

**6. Touch-зоны — минимум 44×44px** для кнопок "Купить", лайков, иконок навигации.

**7. Горизонтальные списки (категории, сторис, фильтры) — скролл, не обрезание:**
```css
overflow-x: auto;
-webkit-overflow-scrolling: touch;
scrollbar-width: none;
```

**8. Навигация по устройствам (см. PLAN.md):**
- Desktop/Laptop: левое меню + верхняя панель + правая информационная панель
- Tablet: компактное меню, без правой панели
- Mobile: Bottom Navigation + Floating Action Button, свайпы для видео-ленты и Live Shopping

**9. Формы (чекаут, создание товара) — одна колонка на мобильных:**
```html
<div class="grid grid-cols-1 md:grid-cols-2 gap-4">
```

**10. iOS auto-zoom на инпутах — избегать:**
```css
input, select, textarea { font-size: 16px; } /* на мобильных брейкпоинтах */
```

#### Чек-лист перед каждым новым UI-компонентом
```
[ ] Проверено на 375px (мобильный)
[ ] Проверено на 768px (планшет)
[ ] Проверено на 1440px (Laptop)
[ ] Проверено на 1920px+ (Desktop)
[ ] Горизонтального скролла body нет
[ ] Touch-зоны ≥ 44px
[ ] Текст не выходит за границы контейнера
[ ] Изображения не растянуты/не обрезаны неправильно
[ ] Использованы CSS-переменные темы (--bg, --surface, --accent и т.д. из CLAUDE.md), не хардкод цветов
[ ] Светлая и тёмная тема — обе проверены
```

## SKILL 3 — СТРУКТУРА ФАЙЛОВ: монорепо TREETEX

### Применять при создании каждого нового модуля, компонента, endpoint'а

#### Правило: один сервис — своя папка, один тип — своё место внутри сервиса

```
treetex/
├── docker-compose.yml     — оркестрация всех сервисов
├── PLAN.md                — техническое задание
├── CLAUDE.md              — описание проекта, стек, токены
├── SKILL.md               — этот файл
├── history/               — журнал значимых решений (см. правило в CLAUDE.md)
├── apps/
│   ├── web/               — Next.js frontend
│   │   ├── app/           — страницы и layout'ы (App Router), группы по ролям: (buyer)/(seller)/(admin)
│   │   ├── components/    — переиспользуемые React-компоненты
│   │   │   └── ui/        — базовые компоненты UI Kit (Button, Card, Modal и т.д.)
│   │   ├── app/globals.css — глобальные стили и CSS-токены
│   │   └── Dockerfile.dev
│   └── api/                — NestJS backend
│       ├── src/
│       │   ├── main.ts, app.module.ts
│       │   └── <feature>/  — по одному модулю на предметную область (products/, orders/, users/, streams/, auth/)
│       │       ├── <feature>.module.ts
│       │       ├── <feature>.controller.ts
│       │       ├── <feature>.service.ts
│       │       ├── entities/
│       │       └── dto/
│       └── Dockerfile.dev
└── packages/
    └── shared/            — общие TypeScript-типы для web и api
        └── src/index.ts
```

#### При добавлении нового модуля — обязательно

| Тип | Куда класть |
|-----------|------------|
| Новая страница (frontend) | `apps/web/app/(роль)/<screen-id>/page.tsx`, где `<screen-id>` — тот же строковый идентификатор экрана, что использовался в прототипах (`feed`, `cart`, `checkout`, `dash`, `overview` и т.д. — см. [002](history/002-sozdanie-docker-infrastruktury.md) и отчёт разведки) |
| Переиспользуемый UI-компонент | `apps/web/components/ui/` |
| Компонент, специфичный для одной роли/экрана | `apps/web/components/<роль>/` |
| Новый backend-модуль (предметная область) | `apps/api/src/<feature>/` со своими controller/service/entities/dto |
| Общий тип данных (Product, Order, User, Stream и производные) | `packages/shared/src/index.ts` (или отдельный файл в `packages/shared/src/`, если список типов вырастет) |
| Миграция базы данных | `apps/api/src/migrations/` (при переходе с `synchronize: true` на явные миграции перед продакшном) |
| Переменные окружения / секреты | `.env` (не коммитить), пример — в `.env.example` |

#### Что запрещено

- Хранить бизнес-логику в компонентах `app/` вместо вынесения в `components/`/сервисный слой
- Смешивать frontend- и backend-код в одном пакете
- Класть новый backend-модуль без структуры `module/controller/service` — плоские файлы в корне `src/` не создавать
- Дублировать типы данных отдельно в `apps/web` и `apps/api` вместо переиспользования `packages/shared`
- Хранить секреты вне `.env`
- Коммитить `node_modules`, `.next`, `dist` (см. `.gitignore`)

#### Именование файлов

```
React-компоненты:  PascalCase.tsx      (ProductCard.tsx, LiveChat.tsx)
NestJS-файлы:      kebab-case.ts       (products.controller.ts, products.service.ts)
Страницы App Router: page.tsx, layout.tsx (фиксированные имена Next.js)
CSS:               kebab-case.css или globals.css
Конфиги:           kebab-case или с точкой (.env, docker-compose.yml)
Записи истории:    NNN-краткое-описание.md (см. правило в CLAUDE.md)
```

#### Чек-лист при каждом новом модуле
```
[ ] Файл лежит в правильном apps/* или packages/* по назначению
[ ] Имя файла соответствует конвенции (PascalCase/kebab-case)
[ ] Backend-модуль имеет module/controller/service/dto, а не плоский файл
[ ] Общие типы — в packages/shared, не продублированы
[ ] Нет секретов вне .env
[ ] CLAUDE.md обновлён, если появилась новая значимая папка/сервис
```

## SKILL 4 — ДОСТУП К ДАННЫМ: только через обёртки, никогда напрямую в разметке

### Применять к каждому новому React-компоненту, странице `app/`, любому `.tsx`-файлу, который выводит данные пользователю

### Суть правила

**Публичный/пользовательский HTML/JSX не должен содержать "сырого" кода доступа к данным.** Внутри разметки (то, что рендерится в `return (...)` компонента) не может быть:
- прямого чтения поля сущности (`product.price`, `order.status`, `user.email`, `stream.viewersCount` и т.д.);
- прямого вызова `fetch`, `api.get/post/patch/delete`, `axios`, `XMLHttpRequest` внутри тела компонента или прямо в JSX;
- инлайновых вычислений над данными сущности (форматирование цены, перевод статуса на человеческий язык, вычисление скидки, склонение слов) прямо в JSX-выражении `{...}`.

Вместо этого разметка обращается к данным **только через обёртку** — компонент или хук, который сам знает, откуда взять данные (движок/API) и как их показать. Обёртка — это единственное место, где написан код; сама разметка содержит только вызов обёртки, то есть выглядит как тег, а не как код.

### Почему это правило существует

Это прямое продолжение архитектуры дизайн-прототипов проекта (`.dc.html` + `support.js`), где разметка использовала плейсхолдеры вида `{{price}}`, `{{product.name}}` — сам HTML не знал, откуда берутся данные и как они форматируются, это делал рантайм-движок `support.js`. При переходе на реальный код эту же дисциплину продолжают выполнять TypeScript-хуки и функции-обёртки: `{{price}}` из прототипа концептуально равен вызову `<Price value={product} />` или `{formatPrice(product.price)}` в реальном коде — разметка не должна знать больше, чем прототип знал.

Практическое обоснование (уже был инцидент в этом проекте): при переходе со старых mock-данных на реальный API несколько страниц (`live`, `video`) остались забыты и продолжали дёргать старую функцию `imgUrl(seed)` с полем `seed`, которого больше не существовало в реальной сущности `Product` — потому что обращение к данным было размазано по разметке вместо того, чтобы быть сосредоточено в одной обёртке, которую было бы видно и легко найти при рефакторинге.

### Терминология: что считается "обёрткой"

Обёртка — это одно из двух:

1. **Хук данных** (`use<Entity>`, `use<Entity>s`) в `apps/web/lib/data/use<Entity>.ts` — единственное место, которое вызывает `api.get/post/patch/delete`. Возвращает уже готовые данные (`{ data, loading, error }` или похожую форму). Компонент вызывает хук и получает результат — сам не знает, что за этим стоит HTTP-запрос.
   Пример: `useProducts()`, `useOrders()`, `useSellerStats()`, `useCmsPages()`.

2. **Функция-форматтер / компонент-обёртка** — берёт сырое поле сущности и возвращает готовое к показу значение или JSX. Живёт рядом с соответствующим хуком данных, экспортируется из того же файла в `apps/web/lib/data/`.
   Пример: `toCardData(product)`, `productImgUrl(product)`, `productBadge(product)`, `orderStatusLabels[status]`, `formatPrice(price)`.

### Нотация обёртки в разметке — `[tag]`

При разметке JSX новый смысловой узел, который является обёрткой (а не сырым полем), должен быть узнаваем "с одного взгляда" — так же, как в исходных `.dc.html`-прототипах плейсхолдер `{{price}}` был визуально отличим от обычного текста. Поэтому:

- Каждая обёртка **обязана** оформляться как компонент с PascalCase-именем, используемый как настоящий JSX-тег: `<Price value={product} />`, `<OrderStatus order={order} />`, `<ProductImage product={product} />` — то есть в квадратных скобках синтаксиса JSX это выглядит как `<Tag ... />`, а не как `{product.price}` россыпью внутри текста.
- Если обёртка — просто функция-форматтер (не JSX-компонент, а строка/число), она вызывается один раз в отдельном выражении `{formatPrice(product.price)}`, а не через цепочку промежуточных вычислений внутри самого JSX.
- Правило считается нарушенным, если в `return (...)` компонента встречается точка сразу после имени сущности с бизнес-полем вне вызова обёртки: `product.price`, `order.status`, `user.rating` — это сигнал "здесь сырой код там, где должен быть тег".

### Что можно / нельзя — таблица

| Ситуация | ❌ Нельзя (сырой код в разметке) | ✅ Можно (через обёртку) |
|---|---|---|
| Показ цены | `<span>{product.price} ₴</span>` | `<span>{formatPrice(product.price)}</span>` |
| Карточка товара | `<img src={\`https://picsum.photos/seed/${product.imageSeed}\`} />` | `<img src={productImgUrl(product)} />` |
| Статус заказа | `<span>{order.status}</span>` | `<span>{orderStatusLabels[order.status]}</span>` |
| Загрузка списка | `useEffect(() => { fetch("/api/products").then(...) }, [])` внутри страницы | `const { products } = useProducts();` |
| Рейтинг | `{product.rating.toFixed(1)}` инлайн в JSX | `<Rating value={product.rating} />` либо форматтер из `lib/data/` |
| Скидка/бейдж | `{Math.round((1 - price/compareAtPrice) * 100)}%` инлайн | `{productBadge(product)}` |

### Область действия

- Применяется ко всем файлам в `apps/web/app/**/page.tsx`, `apps/web/app/**/layout.tsx` и любым `.tsx` в `apps/web/components/`.
- Не применяется к самим обёрткам (`apps/web/lib/data/*.ts`) — там как раз и живёт код обращения к данным, это разрешённое и единственное место для него.
- Не применяется к чисто презентационным пропсам без связи с сущностью бэкенда (`className`, `style` для анимаций, локальный UI-стейт компонента типа `useState` для открытия модалки) — правило касается именно данных, приходящих из движка/API, а не UI-механики.

### Чек-лист перед каждым новым компонентом/страницей
```
[ ] В return(...) нет прямого чтения бизнес-поля сущности (product.X, order.X, user.X) вне вызова обёртки
[ ] В теле компонента нет fetch/api.*/axios — только вызов use<Entity>() хука
[ ] Форматирование (цена, дата, статус, скидка, URL картинки) — вызов функции из lib/data/, не инлайн-выражение
[ ] Новая сущность с backend → сразу заведён use<Entity> хук в apps/web/lib/data/, даже если пока один потребитель
[ ] Если один и тот же расчёт нужен в двух местах — вынесен в обёртку, а не скопирован
```

Полное обоснование и практика применения также зафиксированы в [ENGINE.md §7.1](ENGINE.md).

## ПРИМЕНЕНИЕ

Claude Code применяет все четыре SKILL автоматически:

- При написании **любого нового endpoint'а** → чек-лист SKILL 1 (безопасность)
- При написании **любого нового UI-компонента/страницы** → чек-лист SKILL 2 (адаптив)
- При **создании нового модуля или файла** → чек-лист SKILL 3 (структура)
- При написании **любой разметки, показывающей данные сущности** → чек-лист SKILL 4 (доступ к данным только через обёртки)
- При **code review** → проверять все четыре чек-листа
- **Не спрашивать** "нужна ли адаптивность" — она нужна всегда (см. PLAN.md: Desktop/Laptop/Tablet/Mobile)
- **Не спрашивать** "нужна ли проверка безопасности" — она нужна всегда
- **Не спрашивать** "куда положить файл" — SKILL 3 определяет структуру
- **Не спрашивать** "можно ли обратиться к полю сущности напрямую в JSX" — нельзя никогда, SKILL 4 обязателен без исключений
- Отдельно от этого файла: после каждого одобренного через `ExitPlanMode` плана — запись в `history/` (правило описано в CLAUDE.md)
