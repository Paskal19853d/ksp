# 001 — Выбор стека: Docker + Next.js + NestJS + PostgreSQL

Дата: 2026-07-08

## Контекст

Проект TREETEX существовал как 4 интерактивных дизайн-прототипа в проприетарном `.dc.html`-формате (UI Kit, Покупець, Продавець, Адмін) с общим рантаймом `support.js`, рендерящимся только внутри конкретного дизайн-инструмента. PLAN.md формулирует итог как дизайн-проект, "готовый для реализации в Figma или последующей разработке фронтенда" — то есть открытым оставался вопрос, какой из этих двух путей выбрать дальше.

Первая рекомендация (в рамках отдельного plan mode) была — довести дизайн до готовности в Figma, отложив техническую реализацию. Пользователь затем передумал и явно попросил перейти к технической разработке в Docker.

## Решение

Стек выбран пользователем через серию уточняющих вопросов:
- **Frontend**: Next.js (React) — естественное продолжение того, что `support.js` уже поднимал React через CDN
- **Backend**: Node.js/NestJS + PostgreSQL — единый язык (TypeScript) на весь стек, вместо альтернативы PHP/Laravel + MySQL
- **Инфраструктура**: Docker (Docker Compose), с целью "полноценный стек (frontend + backend + БД)", а не просто контейнер для старых `.dc.html`-файлов или голый фронтенд-only setup
- **Подход к существующему коду**: максимально перенести разметку/стили/токены из `.dc.html`, а не переписывать с нуля вслепую

Разведка перед принятием решения (агентский анализ всех 4 файлов) показала, что перенос реалистичен: CSS-токены идентичны байт-в-байт во всех файлах, синтаксис `sc-if`/`sc-for`/`{{ }}` почти механически превращается в JSX, состояние приложений — плоский `state` с `go(screen)`-роутером, экраны именованы понятными строковыми ID.
