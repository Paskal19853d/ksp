# 003 — Docker-фиксы и доводка вёрстки покупателя/продавца

Дата: 2026-07-09

## Контекст

После пересоздания WSL VM Docker Desktop не поднимался, стек был не запущен. Далее по ходу сессии пользователь итеративно указывал на конкретные баги во вёрстке ("блок разбросан, как без стилей") и просил продолжать доводку экранов покупателя и продавца по пунктам аудита, без формального прохождения через plan mode — работа велась прямыми короткими запросами ("делай", "пробуй", "далее по списку").

## Решение

Чинить проблемы по мере обнаружения и продолжать закрывать пробелы, выявленные агентским аудитом (`Explore`) по сравнению текущей вёрстки с `.dc.html`-прототипами и PLAN.md: недостающие страницы и нерабочие ссылки у покупателя и продавца. Приоритеты (какие экраны делать в каком порядке) уточнялись у пользователя через явные вопросы на каждом шаге.

## Изменения

**Docker:**
- Docker Desktop не был запущен после пересоздания WSL VM — запущен вручную, дождались готовности daemon.
- Добавлен корневой `.dockerignore` (`node_modules/`, `.next/`, `dist/`, `.git/`) — до этого сборка `web` гоняла в контекст билда `.next` (>1 ГБ), что вызывало сбой `COPY apps/web ./apps/web: not found` (баг buildkit на Windows при большом контексте).
- Обнаружена и обойдена нестабильность параллельной сборки `docker compose build` (гонка кэша buildkit) — сборка `web` и `api` образов по отдельности решает проблему.
- Обнаружено: Turbopack dev-сервер в контейнере `web` не всегда подхватывает **новые** директории роутов App Router через file watcher на смонтированном Windows-volume — стабильно требуется `docker compose restart web`, иногда дважды подряд, если файл дозаписан уже после старта контейнера.
- Сделаны файловые бэкапы: `pg_dump` базы (была пуста — миграций ещё нет) и `docker save` образов `ksp-api`/`ksp-web` в `/backups`.

**Фронтенд — исправленные баги:**
- `tailwind.config.ts` — в проекте массово использовался полушаговый spacing (`4.5, 5.5, 6.5…`), которого нет в дефолтной шкале Tailwind 3.4.4 — классы вроде `p-4.5` не генерировали CSS вообще. Добавлены недостающие значения в `theme.extend.spacing`, что разом починило вёрстку в ~22 файлах.
- `components/seller/ProductDrawer.tsx` — слоты фото товара были декоративными (без `<input type="file">`). Добавлена реальная загрузка через `URL.createObjectURL`, превью, удаление, лимит 4 фото.
- `app/(buyer)/shop/[id]/page.tsx` — страница магазина игнорировала параметр `id`, всегда показывала один и тот же захардкоженный магазин (TechnoHub). Введён единый источник данных `shops` в `lib/data/products.ts` (`shopByName`), страница теперь реально резолвит магазин по id.
- Поправлены 3 места с захардкоженным `/shop/1` (`RightPanel.tsx`, `chat/page.tsx`, `product/[id]/page.tsx`) — теперь ведут на реальный магазин.

**Фронтенд — новые страницы покупателя** (меню профиля вело в никуда — пункты были без `href`/`onClick`):
- `/favorites`, `/subscriptions`, `/coupons`, `/wallet`, `/security`, `/settings` — все на основе существующих моков и общего `AppStateContext` (`favs`, `theme` и т.д.), без нового backend API.

**Фронтенд — новые страницы продавца:**
- `/seller/followers`, `/seller/settings` — доступна через клик по карточке магазина в сайдбаре.
- `/seller/returns` — обработка возвратов (approve/reject), добавлено поле `returnReason`/`returnDecision` в `SellerOrder` и `resolveReturn` в `SellerStateContext`.
- `/seller/reviews` — модерация отзывов с ответом продавца, добавлен `sellerReviews` и `replyToReview` в `SellerStateContext`.
- `/seller/taxes` — справка по налогам ФОП (статическая, без интерактивного состояния).
- `components/seller/SellerNav.tsx` — расширен `navDef` и `extraTitles` под все новые разделы.

## Результат проверки

Все новые и изменённые страницы проверены вживую через `curl` на реальном dev-сервере (не только typecheck): проверялся HTTP-код 200 и наличие ожидаемого контента в отданном HTML (имена магазинов по разным id, тексты отзывов, суммы возвратов и т.д.). Ошибок и warning в логах `docker compose logs web` после каждого шага не обнаружено.
