# 005 — ENGINE.md: архитектура backend-движка

Дата: 2026-07-11

## Контекст

Проект TREETEX к этому моменту имел полностью свёрстанные и адаптивные интерфейсы всех 5 ролей (покупатель, продавец, блогер, админ, компания), но backend был реализован только для авторизации — всё остальное (товары, заказы, магазины, live-эфиры, чат, модерация, CMS) существовало исключительно как mock-данные в клиентском React state.

Пользователь, разбирая раздел CMS в админ-панели, обнаружил, что это визуальная обёртка без реальной логики (захардкоженный список страниц, переключатель "Опубліковано" меняющий только React state, кнопка "Редагувати" вообще без `onClick`), и задал более широкий вопрос: на чём должен быть построен "движок — сердце проекта", учитывая, что цель — не MVP, а платформа на **сотни тысяч – миллионы пользователей** в первый-второй год.

## Решение

Прежде чем писать какой-либо код, было решено зафиксировать архитектурные решения в отдельном постоянном документе `ENGINE.md` в корне репозитория — читается наравне с CLAUDE.md/PLAN.md/SKILL.md в каждой сессии, не одноразовый план на исполнение.

Для проработки конкретных решений был запущен агент-архитектор (Plan-агент), изучивший текущий код (`apps/api/src/auth`, `apps/api/src/users`, mock-shapes в `apps/web/lib/data/*.ts`, docker-compose.yml) и вернувший обоснованные рекомендации по каждому компоненту. Ключевые решения:

- **Модульный монолит** на NestJS, не микросервисы — микросервисы решают организационную проблему (независимые команды), а не проблему масштаба; для соло-разработчика это чистые накладные расходы без выгоды.
- **PostgreSQL остаётся ядром данных** без преждевременного шардинга/реплик — миграции вместо `synchronize: true`, PgBouncer для пулинга; реплики/шардинг — только при конкретных измеримых триггерах.
- **Redis** — единая точка для кэша, throttling между инстансами, pub/sub для live-чата/счётчиков зрителей, очередей (BullMQ).
- **Live-shopping real-time**: WebSocket (`@nestjs/websockets` + `socket.io-redis-adapter`); критический инвариант — списание остатков товара всегда через ACID-транзакцию в Postgres, Redis — только кэш отображаемого значения, никогда источник истины.
- **Видео-инфраструктура**: управляемый сервис (Cloudflare Stream, запасной Mux), явно отклонена идея самостоятельного RTMP/HLS — несоразмерная операционная нагрузка для одного разработчика.
- **Поиск**: Postgres full-text search сейчас; Typesense/Meilisearch (не Elasticsearch) — только при конкретном триггере роста каталога.
- **Медиа-хранилище**: S3-совместимое (Cloudflare R2) + CDN, pre-signed upload URLs; MinIO самохостинг отклонён.
- **CMS**: собственный модуль `cms` в NestJS с блочным `jsonb`-контентом, а не headless CMS-продукт (Strapi/Payload/Directus) — несоразмерная сложность ради нескольких статичных страниц.
- **Границы 17 модулей NestJS** по шаблону уже существующих `auth`/`users` (module/controller/service/entities/dto).
- **Дорожная карта**: миграции → Products → Media → Orders/Cart → Reviews → Seller dashboard API → real-time (Redis+Streams+Chat) → Search → CMS+Marketing → Moderation → Finance → Blogger → Notifications.
- Явный список отложенных решений с триггерами пересмотра (Kubernetes, микросервисы, шардинг, Elasticsearch, своя видео-инфраструктура, headless CMS, MinIO, Kafka, мультирегиональность).

## Изменения

- Создан `ENGINE.md` в корне репозитория — 11 разделов (общая форма архитектуры, слой данных, кэширование/real-time, видео-инфраструктура, поиск, медиа-хранилище, CMS, границы модулей, дорожная карта, что не строить сейчас, источники)
- CLAUDE.md, PLAN.md, SKILL.md не изменялись — ENGINE.md ссылается на них, не дублирует

## Результат проверки

Не применимо — этап архитектурного решения и документирования, без изменений кода. Реализация (начиная с миграций и Products API) — предмет последующих сессий.
