# 004 — Админ-панель и роли блогер/компания

Дата: 2026-07-09

## Контекст

После доводки вёрстки покупателя и продавца ([003](003-docker-fixy-i-dovodka-verstki-pokupatel-prodavec.md)) в PLAN.md оставались нереализованными: полная административная панель (14 разделов) и две роли, упомянутые в ТЗ только как перечисление участников без детализации — блогер и компания. Работа велась без формального plan mode, итеративными запросами пользователя ("что дальше по списку"), с уточняющими вопросами перед каждым архитектурным решением.

## Решение

**Админ-панель.** Реализовать все 14 разделов из PLAN.md по аналогии со структурой продавца (свой `Shell`/`Nav`/`StateContext`), с сайдбаром на все разделы сразу — нереализованные показывались как заглушка `ComingSoon`, затем поэтапно наполнялись реальным функционалом.

**Роль блогера.** Обсуждались две модели: (а) креатор без своих товаров, монетизация через партнёрские ссылки/% с продаж чужих товаров, (б) продавец + контентные вкладки. Выбрана модель (а) — полностью независимая роль с собственной панелью, без товаров/заказов.

**Роль компании.** Обсуждались модели: (а) мульти-аккаунт с командой (менеджер/оператор/аналитик) поверх обычного продавца, (б) B2B опт. Выбрана модель (а) — расширение существующего `SellerShell` разделом "Команда", а не отдельная роль с нуля.

## Изменения

**Админ-панель:**
- `lib/data/admin.ts`, `lib/store/AdminStateContext.tsx` — состояние и моки для пользователей, продавцов, комиссий, категорий, банеров, рекламы, CMS, live-трансляций, настроек платформы, очередей модерации

**Роль блогера:**

## Результат проверки

Каждая новая страница проверялась вживую через `curl` на dev-сервере: HTTP-код 200/307 и наличие ожидаемого контента в отданном HTML (реальные имена, суммы, статусы из моков), после каждого функционального изменения — проверка `docker compose logs web` на ошибки/warning.

Отдельно зафиксирована повторяющаяся особенность окружения: Turbopack dev-сервер в контейнере `web` не всегда подхватывает **новые** директории роутов App Router через file watcher на смонтированном Windows-volume — стабильно требуется `docker compose restart web`, при большом количестве одновременно созданных файлов иногда требовалось 2-3 рестарта подряд.

После завершения работы созданы резервные копии: `docker save` образов `ksp-web`/`ksp-api` (пересобранных с итоговым кодом) и `pg_dump` базы данных (пустая — backend всё ещё каркас без реальных сущностей) в `/backups`.
